\section{Background}
\label{sec:background}

This section introduces preliminaries required to reason about CUAs in a unified manner. We formalize the CUA interaction loop, clarify environment assumptions, and motivate why architectural and lifecycle views are both necessary.

\subsection{Problem Formulation: GUI Interaction as Partially Observable Control}
A CUA operates over a GUI-driven environment $\mathcal{E}$ (web page, desktop OS, mobile app) under a user instruction $g$ expressed in natural language. At step $t$, the agent receives an observation $o_t$ (e.g., screenshot pixels, DOM/A11y tree, or a hybrid representation) and produces an action $a_t$ drawn from an action space $\mathcal{A}$ (mouse/keyboard/touch events, API calls, code execution, or macro-actions). The environment transitions and returns a new observation.
In realistic settings, the interaction is \textbf{partially observable} and \textbf{non-stationary}: interface states can change due to asynchronous events (pop-ups, ads, network updates)~\cite{zhou2024webarena,he2024webvoyager}, while the agent's observation may omit latent states (hidden UI layers, permission dialogs, or system-level context)~\cite{xie2024osworld}.

This differs from classical embodied control in two ways:
\begin{itemize}
  \item GUIs are semantically dense---small visual elements (icons, toggles) can represent high-impact operations.
  \item The environment is often \textbf{adversarial or untrusted}: web content, documents, and tool outputs can embed instructions intended to override user intent (prompt injection), motivating explicit trust boundaries in agent design~\cite{liu2025wainjectbench,robison2025aegis}.
\end{itemize}

\subsection{Environment Modalities and State Representations}
A central design axis is how state is represented for reasoning and grounding.
In web settings, DOM trees and accessibility metadata provide structured, machine-readable UI descriptions; this supports deterministic element selection but can be fragile across platforms and fails in rendering-heavy or remote/virtual environments~\cite{zhou2024webarena}.
Conversely, pixel-based perception treats GUI interaction as a vision-language grounding problem, enabling platform-agnostic deployment at the cost of localization errors and sensitivity to resolution, scaling, and visual perturbations.
Hybrid representations combine vision with structural priors (OCR, layout graphs, hierarchical parsing), seeking robust grounding without full dependency on system APIs.

From a benchmarking perspective, Mind2Web targets generalist web instruction following on diverse websites~\cite{deng2023mind2web}, WebArena provides realistic and reproducible web environments~\cite{zhou2024webarena}, and OSWorld evaluates open-ended tasks in real computer environments spanning arbitrary applications~\cite{xie2024osworld}. These benchmarks collectively motivate a unified taxonomy for comparing agents operating under different observation channels.

\subsection{Action Spaces and Execution Interfaces}
The \textbf{action space} determines not only capability but also the safety surface area.
Low-level atomic actions (click, drag, type) are maximally general but error-prone: small coordinate offsets can cause misclicks, and delayed execution can lead to time-of-check/time-of-use mismatches.
Higher-level actions (API invocation, code-as-action, macro-actions) improve efficiency and auditability but introduce new risks, including over-privilege and tool misuse.
Modern systems increasingly mix action modalities, combining GUI actions with programmatic execution to accelerate workflows.

\subsection{Why Both Architecture and Lifecycle Views Matter}
A purely component-wise taxonomy is insufficient because many failure modes are \textbf{stage-dependent}.
Training-time data and objectives constrain what the agent can perceive (\emph{Creation}), deployment integrations determine what it is allowed to access (\emph{Deployment}), long-horizon execution triggers compounding errors and injection attacks (\emph{Execution}), and updates plus ecosystem interactions drive regression and propagation risk (\emph{Maintenance}).
Accordingly, we adopt a four-stage lifecycle taxonomy---\emph{Creation, Deployment, Execution, and Maintenance}---to align engineering choices with where vulnerabilities originate and how they persist.

\subsection{Security and Governance Preliminaries: Attribution and Trust Boundaries}
To discuss safety in CUAs, we require an attribution lens that distinguishes \emph{why} unsafe behavior occurs.
We build on a threat-modeling view and distinguish curiosity-driven overreach from malice induced by adversarial content, while also recognizing environment-driven harms (e.g., UI deception) that do not require model intent~\cite{raza2026agenticsecurity,kumar2025safearena,kuntz2025harm}.
Empirically, safety benchmarks such as OS-Harm and SafeArena highlight that dialogue-level alignment does not automatically transfer to action-level safety for web/OS agents~\cite{kuntz2025harm,kumar2025safearena}.
