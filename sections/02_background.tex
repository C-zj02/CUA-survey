\section{Background}
\label{sec:background}

This section introduces the technical preliminaries needed to reason about computer-use agents (CUAs) in a unified way. We first formalize GUI interaction as a partially observable control problem, then summarize the main environment modalities and action interfaces that shape agent behavior in web, desktop, and mobile settings.

\subsection{Problem Formulation: GUI Interaction as Partially Observable Control}
A CUA operates in a GUI-driven environment $\mathcal{E}$, such as a web page, desktop operating system, or mobile application, under a user instruction $g$ expressed in natural language. At step $t$, the agent receives an observation $o_t$---for example, screenshot pixels, a DOM/accessibility tree, or a hybrid representation---and produces an action $a_t$ from an action space $\mathcal{A}$, such as mouse, keyboard, or touch events, API invocations, code execution, or macro-actions. The environment then transitions to a new state and returns a subsequent observation.

In realistic settings, this interaction is naturally \textbf{partially observable} and often \textbf{non-stationary}. Interface states may change due to asynchronous events such as pop-ups, advertisements, background refreshes, or network updates~\cite{zhou2023webarena,he2024webvoyager}. At the same time, the agent's observation may omit latent but decision-relevant state, including hidden UI layers, permission prompts, or system-level context~\cite{xie2024osworld}.

Compared with classical embodied control, GUI-based interaction has two distinctive properties. First, graphical interfaces are semantically dense: small visual elements such as icons, toggles, and menu entries may correspond to high-impact operations. Second, the environment may contain \emph{untrusted or externally controlled content}. Web pages, documents, and tool outputs can introduce instructions or signals that conflict with the user's objective, which makes trust boundaries and mediation mechanisms central to agent design~\cite{liu2025wainjectbench,adapala2025aegis}. This formulation motivates the architectural decomposition discussed later in the paper.

\subsection{Environment Modalities and State Representations}
A central design axis for CUAs is how the environment state is represented for reasoning and grounding.

In web settings, DOM trees and accessibility metadata provide structured and machine-readable descriptions of interface elements. Such representations support deterministic element selection and programmatic control, but they can be brittle across platforms and are less effective in rendering-heavy, remote, or virtualized environments~\cite{zhou2023webarena}. By contrast, pixel-based perception treats GUI interaction as a vision-language grounding problem. This improves platform generality, but it also introduces sensitivity to localization errors, screen resolution, scaling, and visual perturbations. Hybrid approaches combine visual perception with structural priors such as OCR, layout graphs, or hierarchical parsing, aiming to improve grounding robustness without relying fully on system APIs.

These representation choices directly influence both capability and failure behavior. Structured state tends to favor precise element selection, while perceptual state tends to favor broader deployment coverage. For this reason, benchmark design has also diversified across observation channels. Mind2Web targets generalist web instruction following over diverse websites~\cite{deng2023mind2web}, WebArena provides realistic and reproducible web environments~\cite{zhou2023webarena}, and OSWorld evaluates open-ended tasks in real computer environments spanning arbitrary applications~\cite{xie2024osworld}. Together, these benchmarks motivate a unified view of CUAs across heterogeneous interface modalities.

\subsection{Action Spaces and Execution Interfaces}
The \textbf{action space} of a CUA shapes both what the agent can accomplish and how failures manifest in practice.

Low-level atomic actions such as clicking, dragging, scrolling, and typing are highly general, since they can be applied to a wide range of interfaces. However, they are also prone to execution errors: small coordinate offsets may cause misclicks, and delayed execution may introduce time-of-check/time-of-use mismatches when the interface changes between decision and action. Higher-level actions, including API invocations, code-as-action, and reusable macro-actions, can improve efficiency, abstraction, and auditability. At the same time, they may introduce different risks, such as excessive privilege, unsafe tool invocation, or error amplification through automation.

Modern CUA systems increasingly combine multiple action interfaces rather than relying on a single modality. This design choice expands the operational envelope of the agent, but it also broadens the space of possible failure modes. Accordingly, action abstractions should be understood not only as a capability choice, but also as a factor that shapes the system's safety and control surface.
